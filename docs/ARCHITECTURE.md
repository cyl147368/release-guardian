# Release Guardian 架构文档

## 概述

Release Guardian 采用分层架构设计，每层职责清晰，便于测试和维护。

## 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│                     客户端层                                  │
│  Web 控制台 (SPA)  │  API 客户端  │  Webhook 消费者           │
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTP / WebSocket
┌───────────────────────────┴─────────────────────────────────┐
│                     传输层                                    │
│  HTTP 服务器  │  WebSocket 服务器  │  中间件管道              │
├─────────────────────────────────────────────────────────────┤
│                     路由层                                    │
│  静态文件  │  API 端点  │  健康检查  │  指标端点              │
├─────────────────────────────────────────────────────────────┤
│                     业务逻辑层                                │
│  发布服务  │  审批引擎  │  风险评估  │  SLA 监控              │
├─────────────────────────────────────────────────────────────┤
│                     基础设施层                                │
│  持久化  │  审计日志  │  指标采集  │  Webhook 分发           │
└─────────────────────────────────────────────────────────────┘
```

## 核心组件

### 1. HTTP 服务器 (`src/bootstrap.js`)

使用 Node.js 内置 `node:http` 模块，零第三方依赖。

**职责**：
- 监听端口和处理连接
- 优雅关闭（SIGTERM/SIGINT）
- 中间件管道组装

### 2. 中间件管道 (`src/lib/middleware.js`)

洋葱模型，从外到内执行：

1. **指标收集中间件** - 记录请求延迟和状态码
2. **安全头中间件** - 添加安全响应头
3. **CORS 中间件** - 跨域资源共享
4. **内容类型验证** - 检查 POST/PUT/PATCH 的 Content-Type
5. **请求体限制** - 防止大请求攻击
6. **API 密钥认证** - 可选的身份验证
7. **速率限制** - 滑动窗口限流
8. **请求日志** - 结构化日志记录

### 3. 路由层 (`src/app.js`)

基于 URL 模式的简单路由，支持：
- 精确路径匹配
- 正则表达式参数提取（如 `/api/releases/:id`）
- 静态文件服务（`public/` 目录）

### 4. 业务逻辑 (`src/services/releaseService.js`)

**核心功能**：
- 发布生命周期管理
- 风险评分计算（0-100 分）
- 审批路由（基于风险和服务层级）
- SLA 监控和升级
- 窗口冲突检测
- 证据包生成

**风险评分公式**：
```
基础分 = 服务层级分 + 环境分 + 变更类别分
控制分 = (1 - 通过控制数/总控制数) × 30
影响分 = min(客户影响分 × 4, 20) + min(数据敏感度分 × 4, 20)
总分 = 基础分 + 控制分 + 影响分
```

### 5. 持久化 (`src/repository.js`)

JSON 文件存储，特性：
- 原子写入（写入临时文件后重命名）
- 启动时从文件加载
- 支持团队配置

### 6. 审计日志 (`src/lib/audit.js`)

内存审计日志，特性：
- 自动生成唯一 ID 和时间戳
- 支持多维度查询（事件类型/操作者/资源类型/时间范围）
- 自动裁剪防止内存溢出（默认最大 10,000 条）

### 7. 指标采集 (`src/lib/metrics.js`)

Prometheus 兼容指标，包括：
- HTTP 请求计数和状态码分布
- 延迟直方图（P50/P95/P99）
- 业务指标（发布创建/审批/部署数）

### 8. WebSocket 服务器 (`src/lib/websocket.js`)

RFC 6455 标准实现，特性：
- 零第三方依赖
- 事件订阅机制
- 心跳检测（30 秒间隔）
- 二进制帧编解码

### 9. Webhook 引擎 (`src/lib/webhooks.js`)

事件驱动的 Webhook 分发，特性：
- 支持通配符订阅（`*`）
- 异步 HTTP POST 推送
- 事件日志记录

## 数据流

### 创建发布

```
客户端 POST /api/releases
  → 中间件管道（认证/验证/限流）
  → 路由匹配
  → releaseService.createRelease()
    → 验证输入
    → 计算风险评分
    → 判断是否需要人工审批
    → 生成审批链
    → 保存到仓库
    → 写入审计日志
    → 触发 Webhook
  → 返回 201 + 发布数据
```

### 审批流程

```
客户端 POST /api/releases/:id/approvals
  → 中间件管道
  → 路由匹配
  → releaseService.reviewRelease()
    → 验证发布状态为 pending_approval
    → 验证审批团队匹配
    → 更新审批状态
    → 检查是否所有团队都已审批
    → 如果全部通过，更新发布状态为 approved
    → 写入审计日志
    → 触发 Webhook
  → 返回 200 + 更新后的发布
```

## 设计决策

### 1. 零第三方依赖

**决策**：仅使用 Node.js 内置模块。

**原因**：
- 减少供应链攻击风险
- 简化部署和维护
- 避免依赖冲突
- 提高启动速度

**权衡**：
- 需要自己实现一些功能（如 WebSocket）
- 代码量可能增加

### 2. JSON 文件存储

**决策**：使用 JSON 文件而非数据库。

**原因**：
- 简化部署（无需额外服务）
- 便于备份和迁移
- 足够中小规模使用

**权衡**：
- 不适合高并发写入
- 不支持复杂查询

### 3. 内存审计日志

**决策**：审计日志存储在内存中。

**原因**：
- 快速读写
- 简单实现
- 支持实时查询

**权衡**：
- 重启后丢失（可通过持久化扩展）
- 内存使用量有上限

### 4. 洋葱模型中间件

**决策**：使用函数组合的洋葱模型。

**原因**：
- 清晰的执行顺序
- 便于测试每个中间件
- 灵活的管道配置

## 扩展点

### 1. 自定义持久化

实现 `Repository` 接口即可替换 JSON 文件存储：

```javascript
class CustomRepository {
  async load() { /* ... */ }
  async save(data) { /* ... */ }
}
```

### 2. 自定义审计日志

实现审计日志接口即可替换内存实现：

```javascript
class CustomAuditLog {
  record(event, options) { /* ... */ }
  query(filters) { /* ... */ }
  getStats() { /* ... */ }
}
```

### 3. 自定义指标采集

实现指标接口即可替换 Prometheus 实现：

```javascript
class CustomMetrics {
  recordRequest(method, path, statusCode, durationMs) { /* ... */ }
  getSnapshot() { /* ... */ }
  toPrometheus() { /* ... */ }
}
```

## 安全考虑

### 1. 输入验证

所有输入都经过验证：
- 类型检查
- 范围检查
- 枚举值检查
- 字符串清理（HTML 转义、控制字符移除）

### 2. 认证和授权

- 可选的 API 密钥认证
- 白名单路径（健康检查、文档）
- CORS 配置

### 3. 速率限制

- 基于 IP 的滑动窗口
- 可配置的窗口大小和最大请求数
- 返回标准的 RateLimit 头

### 4. 安全头

- X-Content-Type-Options: nosniff
- X-Frame-Options: DENY
- Strict-Transport-Security
- Content-Security-Policy
- Referrer-Policy
- Permissions-Policy

## 性能优化

### 1. 静态文件缓存

- HTML: no-cache（总是验证）
- CSS/JS/图片: public, max-age=3600（1 小时）

### 2. ETag 支持

- 基于 SHA1 的内容哈希
- 支持 If-None-Match 条件请求

### 3. 异步处理

- Webhook 推送异步执行
- 不阻塞主请求处理

## 监控和可观测性

### 1. 结构化日志

JSON 格式日志，包含：
- 时间戳
- 日志级别
- 请求 ID
- 方法和路径
- 状态码
- 响应时间

### 2. Prometheus 指标

标准 Prometheus 格式，包括：
- rg_uptime_seconds
- rg_http_requests_total
- rg_http_request_duration_ms
- rg_releases_created_total
- rg_sla_breaches_total

### 3. 健康检查

- `/health` - 简单存活检查
- `/ready` - 就绪检查（包含数据存储状态）

### 4. 审计日志

所有状态变更都记录审计日志：
- 发布创建/更新/审批/部署
- Webhook 订阅/删除
- 批量操作
